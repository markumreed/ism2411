Today's walkthrough

def, parameters, and return

A function is a chunk of code with a name. You define it once and call it many times. The def keyword starts the definition; the parameter names in parentheses act as placeholders for the values that will be passed in; return sends a result back to wherever the function was called. Without return, the function runs its body and then silently returns None.

Think of a function as a vending machine: you insert inputs (arguments) and get back an output (the return value). The machine doesn't care who uses it — it just processes whatever is put in. This is what makes functions so powerful: you write the logic once and apply it to any data.

The naming convention for Python functions is lowercase with underscores: calculate_tax, not CalculateTax or calculateTax. The name should describe what the function does — usually a verb phrase. A function named result or data tells you nothing; a function named apply_discount is self-documenting.

def calculate_tax(price, rate):
    return price * rate

print(calculate_tax(100, 0.07))   # 7.0
print(calculate_tax(250, 0.08))   # 20.0
print(calculate_tax(0, 0.07))     # 0.0

Reading a traceback

When Python crashes it prints a traceback. Read it bottom-up. The bottom line tells you what went wrong (the error type and message); the lines above show the call chain — which function called which, and on which line. The most useful information is almost always in the last two lines.

Common error types and what they mean: NameError — you used a variable that doesn't exist (check spelling or scope). TypeError — you called a function with wrong argument types or count. SyntaxError — Python can't parse your code (look for missing colons, unmatched brackets). IndexError — you tried to access a list index that doesn't exist. ZeroDivisionError — you divided by zero.

The traceback also shows which file and which line number the error occurred on. In long scripts, this saves enormous time — Python tells you exactly where to look. The habit to build: before Googling, before asking AI, read the traceback. Often the answer is right there in plain English.

Traceback (most recent call last):
  File "sales.py", line 12, in <module>
    total = compute_total(price)
  File "sales.py", line 5, in compute_total
    return price * quantity
NameError: name 'quantity' is not defined

Debug First, Then Ask — the in-class exercise

You will get a script with 3 intentional bugs. Step 1: try to debug using only print() and the traceback. Add print statements before, during, and after suspect code to see what values variables actually hold. This is called print-debugging, and it is the first tool every developer reaches for. Step 2: paste only the error message into ChatGPT or Claude and ask what it means — NOT for the fix. Step 3: fix it yourself, using the AI's explanation as a guide. Step 4: class debrief on what you found.

This sequence matters. If you go straight to "fix my code" in an AI tool, you skip the most valuable part of the process: understanding why the error occurred. The exam will show you error messages and ask you to identify the cause. If you have never debugged code yourself, you will not be able to answer those questions.

The three modes of AI use

Explainer — "What does this error message mean?" or "Explain what a NameError is." Low risk: the AI is teaching you a concept. You still have to understand and apply it. This mode actively supports your learning and is fully acceptable.

Debugger — "Here is my error and my code. Where should I look?" Medium risk: the AI points you toward the problem area. You still have to understand and fix it yourself. Acceptable if you disclose it and can explain the fix.

Generator — "Write this function for me." High risk: if you submit AI-generated code you do not understand, you cannot explain it on the exam or in an interview. You are also at significant academic integrity risk if you do not disclose. Use this mode only for scaffolding — starter code you will read, modify, and fully understand before submitting.

Disclosure

Any time you use an AI tool on graded work, disclose it in a comment at the top of your file. The format: # AI use: [describe what you used and how]. For example: # AI use: asked Claude what NameError means on line 14; fixed it myself. The disclosure protects you and is required by USF Academic Integrity policy. Non-disclosure combined with AI-generated code is treated as academic dishonesty.

Detailed Code Walkthrough

Walkthrough 1: Function composition

Functions become most powerful when composed — the output of one function becomes the input of another. This is exactly how data pipelines work in professional code. Each function does one thing; complex behavior emerges from combining them.

def apply_discount(price, tier):
    """Return discounted price based on customer tier."""
    if tier == "gold":
        return price * 0.85      # 15% off
    elif tier == "silver":
        return price * 0.90      # 10% off
    else:
        return price             # no discount

def calculate_tax(price, rate=0.07):
    """Return price + tax. Default rate is 7%."""
    return price * (1 + rate)

def final_price(original, tier, tax_rate=0.07):
    """Compose discount + tax into a single call."""
    discounted = apply_discount(original, tier)  # Step 1
    total = calculate_tax(discounted, tax_rate)  # Step 2
    return total

# Call the composed function
print(f"${final_price(200, 'gold'):.2f}")   # $179.90
print(f"${final_price(200, 'silver'):.2f}") # $192.60
print(f"${final_price(200, 'none'):.2f}")   # $214.00

Walkthrough 2: Print-debugging step by step

When a function produces unexpected output, add print statements at each step to inspect intermediate values. Remove them once the bug is fixed.

def compute_total(price, qty, discount):
    print(f"DEBUG: price={price}, qty={qty}, discount={discount}")  # step 1
    subtotal = price * qty
    print(f"DEBUG: subtotal={subtotal}")   # step 2
    after_discount = subtotal * (1 - discount)
    print(f"DEBUG: after_discount={after_discount}")  # step 3
    return after_discount

result = compute_total(25, 4, 0.10)
print(f"Final: ${result:.2f}")

# After debugging — remove the print() lines and keep only the logic

Common Errors

ERROR 1 — Calling before defining
# WRONG — function must be defined before it is called
print(greet("Maria"))   # NameError: name 'greet' is not defined
def greet(name):
    return "Hello, " + name

# RIGHT — define first, call after
def greet(name):
    return "Hello, " + name
print(greet("Maria"))
ERROR 2 — Missing return (returns None)
# WRONG — no return statement
def square(x):
    x * x   # computes but throws away the result

result = square(5)
print(result)   # None — not 25!

# RIGHT
def square(x):
    return x * x   # must return the result

result = square(5)
print(result)   # 25
ERROR 3 — Wrong argument count
# WRONG — too few arguments
def add(a, b):
    return a + b
print(add(5))   # TypeError: add() missing 1 required argument: 'b'

# RIGHT — pass all required arguments
print(add(5, 3))   # 8

Discussion Questions

  1. What is the difference between a parameter and an argument? Give a real-world analogy — for example, a recipe uses "ingredients" as parameters and actual food items as arguments.
  2. Why does scope exist? What problems would arise if every variable was global (visible everywhere)?
  3. You encounter a NameError on line 47 of a 100-line script. Walk through the debugging process step by step: what do you check first, second, third?
  4. A classmate says "I just paste my code into ChatGPT when it doesn't work — it always fixes it." What are the risks of this approach, and how would you explain those risks in terms of both learning outcomes and academic integrity?

Key Takeaways
  • def creates the function; calling it by name with arguments actually runs it.
  • Parameters are placeholders; arguments are the real values. The function does not care what you named the variables outside it.
  • Without a return statement, a function returns None — not 0, not an empty string, but None.
  • Read tracebacks bottom-up. The last line is usually the most useful.
  • The three AI modes: Explainer (low risk), Debugger (medium), Generator (high). Always disclose AI use on graded work.
