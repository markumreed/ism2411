Why this matters

Everyone who has ever lost work to a folder full of finalFINAL_v3.py files knows why version control matters. Git makes every snapshot of your code recoverable. GitHub puts those snapshots online so your work is backed up, shareable, and gradeable. But beyond backup, Git is how the entire software industry collaborates — every open-source project, every startup, every enterprise engineering team uses it. Learning Git now means you can contribute to any of them.

From a business analytics perspective, Git solves a problem that spreadsheets cannot: the audit trail. When a client asks "what changed between last month's report and this month's?", a Git repository gives you a precise, timestamped, line-by-line diff. When a "fix" breaks a working script, Git lets you identify exactly which commit introduced the bug — and revert to the last known good state in one command. For compliance-driven industries (finance, healthcare, pharma), this kind of provenance is not optional.

Starting this module, every assignment in this course is submitted via GitHub. That means your code is now public-facing — at least to your instructor and potentially to future employers. A clean, well-documented repository with descriptive commit messages is a professional artifact. It is one of the first things recruiters in technical roles look at. Think of your GitHub profile as your code portfolio: every commit is a data point about how you work.

Core Concepts

Repository (repo)

A folder tracked by Git. Every file change inside it can be recorded, reviewed, and reverted. Think of it as a project folder that also stores the complete history of every version of every file.

commit

A saved snapshot of the repository at a point in time. Every commit has a unique hash (like a fingerprint), an author, a timestamp, and your message. Commits are permanent — you can always go back to any one of them.

Staging (git add)

Before you commit, you tell Git which changes to include using git add. This lets you be selective — you might have 10 changed files but only want to commit 3 of them this time. The staging area is the "waiting room" before the commit.

push / pull

git push uploads your local commits to GitHub (the remote). git pull downloads changes from the remote to your local machine. In team settings, you pull before you push to stay in sync.

.gitignore

A file listing patterns of files Git should not track. Common entries: .env (secrets), __pycache__/ (Python bytecode), .DS_Store (Mac metadata), large data files. Committed to the repo so everyone on the team ignores the same things.

Business Scenario

Your team has been maintaining a Python script that generates a weekly revenue report. Last Tuesday, a colleague "improved" the discount calculation and pushed the change. On Wednesday, the CFO noticed the numbers were wrong — discounts were being applied twice. Without Git, you would be manually comparing file timestamps and hoping someone kept a backup. With Git, you run git log, find the commit from Tuesday, run git diff to see exactly what changed, and either revert that commit or apply a targeted fix.

Here is the workflow that would have prevented — and then fixed — the problem:

# Before making any change, check what's tracked
git status

# See what changed in the last commit
git diff HEAD~1 HEAD

# Revert to the state before Tuesday's commit
git revert abc1234   # abc1234 is the commit hash from git log

# Or check out just one file from a previous commit
git checkout abc1234 -- discount.py

A well-maintained Git history is not just good practice — it is a professional liability safeguard. If a financial model produces the wrong output, being able to prove exactly when and how it changed is the difference between a correctable mistake and an unresolvable audit finding.

Key Vocabulary

  • Git — a version control system that tracks changes to files over time; runs locally on your machine
  • GitHub — a website that hosts Git repositories online; adds collaboration, issue tracking, and code review features
  • Repository (repo) — a project folder tracked by Git, containing all files and their complete history
  • commit — a saved snapshot of the repo; includes a unique hash, message, author, and timestamp
  • Staging area — a holding area for changes you have git add-ed but not yet committed
  • push — uploads local commits to the remote repository (GitHub)
  • pull — downloads changes from the remote to your local machine
  • .gitignore — a file listing file patterns that Git should not track (secrets, build artifacts, OS files)
  • commit message — a short description of what changed; written in imperative tense ("Add tax function", not "Added" or "Adding")
Common Misconceptions

1. "GitHub is Git." No — Git is the version control software that runs on your machine. GitHub is a website that hosts Git repositories. You could use Git without GitHub (just locally), but you cannot use GitHub without Git.

2. "I should commit after everything is perfect." Commits are checkpoints, not final submissions. Commit early and often — at any point where the code is in a consistent, runnable state. Small commits are easier to understand and easier to revert than one giant "added all the things" commit.

3. "git push and git commit are the same thing." No — git commit saves a snapshot locally. git push sends those saved snapshots to GitHub. Many students commit locally for days and then wonder why GitHub doesn't show their work.

Q1 · Multiple Choice
What does git add do?
  • a) Creates a repo
  • b) Stages changes for the next commit
  • c) Sends changes to GitHub
  • d) Shows commit history
Answer
Stages changes for the next commit
Q2 · Multiple Choice
What does git commit -m 'message' do?
  • a) Deletes a file
  • b) Downloads from GitHub
  • c) Saves a snapshot of staged changes with a description
  • d) Creates a branch
Answer
Saves a snapshot of staged changes with a description
Q3 · Multiple Choice
What does git status show?
  • a) All contributors
  • b) Which files are changed and which are staged
  • c) Full commit history
  • d) Python version
Answer
Which files are changed and which are staged
Q4 · Multiple Choice
What is the purpose of a README file?
  • a) Stores the interpreter
  • b) Lists all commits
  • c) Describes the project, how to use it, and relevant notes
  • d) Required to run Python
Answer
Describes the project, how to use it, and relevant notes
Q5 · Code Trace
Put these Git commands in the correct order:
A: git push
B: git add myfile.py
C: git commit -m 'add tax function'
Answer
B → C → A — Stage, then commit, then push
Q6 · Multiple Choice
Which is a good commit message?
  • a) "stuff"
  • b) "fix"
  • c) "add calculate_tax function to main.py"
  • d) "asdfgh"
Answer
"add calculate_tax function to main.py" — Good messages are specific and descriptive
Q7 · Multiple Choice
What is the difference between Git and GitHub?
  • a) They are the same thing
  • b) Git is the website; GitHub is the software
  • c) Git is the version control software; GitHub is a website that hosts Git repositories
  • d) GitHub is only for public projects
Answer
c) Git is the version control software; GitHub is a website that hosts Git repositories
Q8 · Multiple Choice
What should you put in a .gitignore file?
  • a) Your README
  • b) Files that should be tracked by Git
  • c) Files that contain secrets, large data files, and OS artifacts like .DS_Store
  • d) Your Python scripts
Answer
c) Files that contain secrets, large data files, and OS artifacts like .DS_Store
Q9 · Code Trace
A student runs these commands. After the last command, does their code appear on GitHub?
git add sales.py
git commit -m "add sales report"
# student closes laptop without running any more commands
Answer
No — they committed locally but never ran git push. The commit exists on their machine, not on GitHub.
