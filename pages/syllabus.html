Course Description

ISM2411 introduces Python programming through a business analytics lens. Students start from first principles — what a computer is, how the command line works — and build toward loading, cleaning, analyzing, and visualizing real datasets with pandas. The course assumes no prior programming experience and no prior statistics background. Every concept is introduced through business scenarios: pricing formulas, sales transactions, customer records, and retail datasets.

This course is deliberately practical. By the final module, students have a GitHub portfolio with 15 labs and a capstone analysis they can show a recruiter. They also leave knowing when Python is the right tool and when Excel is fine — a judgment most introductory programming courses never teach.

Who this course is for

Business, finance, marketing, supply chain, and management students who work with data in spreadsheets and want to do more. Also appropriate for students who have used no-code tools (Tableau, Power BI) and want to understand what's happening under the hood. Not appropriate for students who already know Python — see ISM3232 or ISM4314 instead.

Learning Outcomes

By the end of this course, students will be able to:

  1. Explain how a computer stores and executes a program; navigate the file system from the command line
  2. Write Python scripts that use variables, conditionals, loops, and functions to automate repetitive business calculations
  3. Read, write, and manipulate structured data using lists, dictionaries, and the csv module
  4. Load datasets from CSV files using pandas, clean messy data, and compute descriptive statistics
  5. Group, filter, and aggregate data with pandas to answer specific business questions
  6. Create bar, line, and histogram charts with matplotlib and seaborn; choose the right chart type for the data
  7. Use Git and GitHub to version-control, organize, and submit code professionally
  8. Use AI coding assistants responsibly — verify outputs, debug errors independently, and disclose AI-generated code per course policy

Lab Grading Rubric

Each lab is scored out of 10 points:

  • 4 pts — Correctness: code produces the expected output for all exercises
  • 3 pts — Completion: all exercises attempted, including stretch challenges (partial credit for reasonable attempts)
  • 2 pts — Code quality: meaningful variable names, no unnecessary repetition, code is readable
  • 1 pt — Submission: submitted on time via the correct method (GitHub repo link or Canvas upload), notebook runs clean from top to bottom

Course Policies

Late Work

Labs are due Sunday at 11:59 pm ET of the module week they are assigned. Late submissions are penalized 10% per calendar day (not per class day) for up to three days. After three days, the lab is not accepted and receives a zero. The late penalty does not apply to the capstone — see the capstone guide for its milestone deadlines.

One free extension

Each student has one 48-hour no-questions-asked extension per semester. To use it, email the instructor before the deadline with the subject line "ISM2411 Extension — Module NN." You cannot use the extension on the midterm or capstone.

Attendance

Attendance is not taken and is not directly graded. However, participation (5% of your grade) is affected by engagement in class. In practice, students who miss more than three classes without notice struggle with labs — the material compounds quickly. If you must miss a session, review the lecture page for that module and complete the in-class activity on your own before the next session.

Quiz Policy

Quizzes are on Canvas, open before Tuesday's session, and due before Thursday's lab. They are open-note and untimed. The purpose is to ensure you've done the reading before lecture — not to test your ability to memorize under pressure. Questions marked ★ MIDTERM-ELIGIBLE or ★ CAPSTONE-ELIGIBLE reappear on formal assessments verbatim or with minor variation. Treat the quiz as your first study session for the exam.

Midterm Makeup

If you cannot attend the midterm due to illness or a documented emergency, email the instructor before the exam starts. Makeup exams are arranged through USF Student Accessibility Services or directly with the instructor for verified emergencies. Skipping the exam without contact results in a zero — there is no default makeup window.

Laptop and Technology

Bring your laptop to every session. During labs, you are expected to have VS Code and Python running. During lecture, use your laptop to follow along with code — but close social media and unrelated tabs. During the midterm, all devices are prohibited except a printed cheatsheet.

Academic Integrity & AI Policy

Collaboration

You may discuss concepts, error messages, and general approaches with classmates. You may not share code for graded labs — each student's submission must be written independently. If you debug together, each of you must write your own solution from scratch afterward. The standard is: can you explain every line of your submission without help? If not, it's not yours.

AI Coding Assistants

This course teaches professional use of AI tools — not avoidance of them. There are three approved modes:

  • Explainer mode: ask AI to explain a concept or a piece of code you don't understand. Always encouraged.
  • Debugger mode: paste an error message and ask AI to explain what went wrong. Allowed, but try to read the traceback yourself first.
  • Generator mode: ask AI to write code for you. Allowed only if you (a) understand every line it produces, (b) can explain it without the AI, and (c) disclose that AI generated it in your submission comment.
Prohibited uses

Using AI to complete the midterm exam in any form is academic dishonesty. Submitting AI-generated code without disclosure is academic dishonesty. Using AI to write code you cannot explain is a waste of this course — you'll pay for it on the midterm and the capstone interview portion.

See the full AI Use Policy page for examples of each mode and the exact disclosure language to use in submissions.

USF Academic Honesty Policy

Violations are reported to the Office of Student Conduct per USF policy (FGCU 6C4-10.081). Confirmed violations result in a zero on the assignment and may result in a failing grade in the course. The policy applies regardless of whether the student knew the rules — read this syllabus.

Resources & Support

Office Hours

Announced in Module 1. Office hours are the highest-ROI activity in this course — bring a specific error or concept question. You will leave understanding something you didn't before.

Student Accessibility Services

Register with USF SAS and provide documentation in the first two weeks. Extended time, alternate testing locations, and other accommodations are arranged through SAS — not at the door on exam day.

USF IT Help Desk

For laptop, network, or software installation issues. Contact before the second class if you're having trouble getting Python or VS Code running. Don't wait until Lab 1 to discover a broken install.

Writing Studio & Tutoring

The capstone requires written interpretation of results. USF's Writing Studio can help with clarity. Programming tutoring is available through the College of Engineering tutoring center (open to all students).

Stuck on a bug?

Before emailing: (1) read the full traceback — the answer is usually in the last two lines, (2) search the exact error message in quotes, (3) check the relevant module's Reading and Lecture pages for a Common Errors section. If you've done all three and are still stuck, bring the error to office hours or post it in the class discussion board with your code pasted as text (not a screenshot).

Course Modifications

The instructor reserves the right to modify the schedule, assignments, and grading breakdown with reasonable advance notice. Any changes will be announced in class and reflected on this site. The core structure — labs, midterm, capstone — will not change after Module 2.
