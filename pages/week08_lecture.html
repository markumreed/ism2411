Today's walkthrough

Why version control

You have changed a working spreadsheet, broken it, and could not get back to the working version. Git fixes that — permanently. Every commit is a snapshot you can return to. Think of commits as save points in a video game: you can always reload from any previous save, and the game keeps a log of every save you ever made.

More than just backup, version control is how you communicate intent. A commit message like "fix off-by-one error in discount tier calculation" tells a future reader (often your future self) exactly what changed and why — much more useful than "updated file" or no message at all. In professional settings, commit messages are part of your code documentation.

For data analytics work specifically, Git solves the reproducibility problem: if a colleague or regulator asks "what does this script do, and how has it changed?", a Git repository gives you a precise, auditable answer. Every line change, every date, every author is permanently recorded.

The four commands you need this semester

You only need four commands to submit assignments in this course. Memorize them in order — they always go in this sequence: initialize once, then for every change you make, stage it, commit it with a message, and push it to GitHub.

git status is your compass — run it before and after every other command to understand where you are. It tells you which files have changed, which are staged, and whether you are ahead of or behind GitHub.

git log shows your commit history: every commit hash, author, date, and message. Use it to review what you have done and to find a specific commit you want to revisit or revert.

git init                          # one-time per repo
git status                        # what changed?
git add file.py                   # stage a file
git commit -m "add tax function"  # save a snapshot
git push                          # send to GitHub

Connecting to GitHub

GitHub is a website that hosts Git repositories. You push your local snapshots up so your work is backed up and gradeable. The connection between your local repo and GitHub is called a "remote" — a named reference to a URL. The conventional name for your primary remote is origin.

Once you have set up the remote once, every subsequent push just needs git push. If you are working on a shared project, always run git pull before pushing to download any changes your collaborators have made. This keeps your local copy in sync.

# one-time per repo: connect local to GitHub
git remote add origin https://github.com/you/ism2411.git

# first push — set upstream tracking
git push -u origin main

# every time after that:
git push

Repo structure for this class

Every weekly submission goes in a folder. This keeps the repo organized and makes it easy for your instructor (and future employers looking at your work) to navigate. A consistent structure also means you never have to think about where to put files.

The README.md at the root level is the front page of your repo. It shows up automatically on GitHub and should describe what the project does, who made it, and how to run the code. Think of it as the cover page of a report — it is the first thing anyone reads.

ism2411/
├── README.md          # describes your project
├── .gitignore         # files to exclude from tracking
├── week05/
│   └── discount.py
├── week06/
│   └── sales_loop.py
├── week07/
│   └── functions.py
└── week08/
    └── main.py

Commit message standards

Good messages describe what changed, written in the imperative tense (as if giving a command): "Add revenue function", not "Added" or "Adding". The subject line should be under 72 characters and answer the question "If applied, this commit will ___." For longer explanations, add a blank line after the subject and write a paragraph.

Bad commit messages are a form of technical debt. When something breaks three months from now and you need to find when a specific behavior changed, messages like "fix", "update", and "stuff" are useless. Messages like "fix off-by-one error in range() call in sales loop" are gold. Your commit history is your work diary — write it so future-you can understand it.

# BAD — tells you nothing
git commit -m "stuff"
git commit -m "fix"
git commit -m "changes"

# GOOD — specific and descriptive
git commit -m "add calculate_tax function"
git commit -m "fix off-by-one error in discount tier"
git commit -m "refactor sales loop to use accumulator pattern"

Detailed Code Walkthrough

Walkthrough 1: The full workflow from scratch

This is the complete sequence you will run when setting up a new project. Each step builds on the previous one. Notice how git status appears twice — it is your checkpoint before and after every significant action.

# Step 1: navigate to your project folder and initialize
# (assumes you're in your terminal)
git init                    # creates a hidden .git folder

# Step 2: create a file
# (create sales.py with your code)

# Step 3: check what Git sees
git status                  # shows: sales.py is "Untracked"

# Step 4: stage the file
git add sales.py            # moves sales.py to staging area

# Step 5: confirm it's staged
git status                  # shows: sales.py is "Changes to be committed"

# Step 6: commit with a meaningful message
git commit -m "add initial sales analysis script"

# Step 7: verify the commit was saved
git log --oneline           # shows one line per commit with hash + message

# Step 8: push to GitHub (after setting up remote)
git push

Walkthrough 2: The everyday update cycle

After the initial setup, your day-to-day workflow is much simpler. Make changes, check what changed, stage, commit, push. This cycle might take 30 seconds once it is habitual.

# You make some changes to functions.py...

git status            # see: functions.py modified

git diff functions.py # see exactly what lines changed (optional but useful)

git add functions.py

git commit -m "add apply_discount function with gold/silver/bronze tiers"

git push              # done — changes are on GitHub

Common Errors

ERROR 1 — Committed but not pushed
# Student commits locally but GitHub shows nothing
git commit -m "add my work"
# Forgot git push!

# Fix: just run push
git push
ERROR 2 — Staged wrong file
# You accidentally staged a file you didn't mean to
git add .env         # oops — this has your passwords!

# Unstage it (does NOT delete the file)
git restore --staged .env

# Then add it to .gitignore so this doesn't happen again
echo ".env" >> .gitignore
ERROR 3 — "Nothing to commit" confusion
# git commit says "nothing to commit"
# This means: either no files have changed, OR
# you changed files but forgot to git add them first

# Diagnose with:
git status          # look for "modified" files that aren't staged

# Fix: stage the changed files, then commit
git add functions.py
git commit -m "update discount logic"

Discussion Questions

  1. A colleague says "I just email my Python files back and forth with the team — what does Git add that email doesn't?" Make the case for Git in terms they would find compelling given their workflow.
  2. Why is "add calculate_tax function" a better commit message than "update file"? What information does each version preserve, and how does that matter in a team context 6 months from now?
  3. What should and should not go in a .gitignore? Give three examples of each, with reasoning.
  4. You discover that a bug was introduced 5 commits ago. With Git, how would you identify which commit is responsible? What command would you use?

In-class activities

  • Create a GitHub account if you do not have one yet.
  • Initialize a repo, add a README, make 3 commits with descriptive messages.
  • Push to GitHub and verify it shows up in the browser.
  • Add a .gitignore file. Include at minimum: .DS_Store, __pycache__/, .env
  • Use git log --oneline to view your commit history and describe what each commit changed.
  • Partner exercise: review each other's commit messages and suggest improvements for any that are vague.

Key Takeaways
  • The core workflow: git add → git commit -m "message" → git push. In that order, every time.
  • git commit saves locally; git push sends to GitHub. Students who only commit often wonder why GitHub shows nothing.
  • Commit messages are documentation. Write them for your future self and your teammates.
  • Use .gitignore to keep secrets, credentials, and build artifacts out of your repo.
  • A clean, well-organized GitHub repo with good commit messages is a portfolio item that employers look at.
