Dictionaries

Dictionaries store related data together with labels. This is the bridge to working with structured business records — customers, products, orders — and a stepping stone to pandas DataFrames in Unit 4.

Today's walkthrough

Key-value pairs

A dictionary is a set of key-value pairs in curly braces. Keys are usually strings; values can be anything — strings, numbers, lists, even other dictionaries. You access a value by its key, not by a position number. This makes code dramatically more readable: customer["tier"] tells you exactly what you're looking up, while customer[3] tells you nothing.

In a business context, think of a dictionary as one row in a database table, where each column name is a key. A CRM entry is a dictionary. A product record from an inventory system is a dictionary. A line item on an invoice is a dictionary. Once you see this pattern, you'll find dictionaries everywhere.

Keys must be unique within a dictionary — you cannot have two entries with the key "tier". If you assign to an existing key, it overwrites the old value. This is how updates work.

customer = {
    "name": "Alice",
    "tier": "gold",
    "ytd_spend": 12400,
}

print(customer["name"])         # Alice
print(customer["ytd_spend"])    # 12400

Adding, updating, deleting

Adding a new key works the same way as updating an existing one — just assign to it. If the key exists, the value is overwritten. If it doesn't exist, a new entry is created. This simplicity is one of Python's strengths: no special method needed, just assignment.

Deleting uses either the del statement (which raises a KeyError if the key is missing) or .pop(key) (which also returns the deleted value and can accept a default to avoid the error). Use .pop() when you want to retrieve the value as you delete it.

customer["email"] = "a@x.com"   # add new key
customer["tier"] = "platinum"     # update existing key
del customer["ytd_spend"]         # delete — raises KeyError if missing
removed = customer.pop("email", None)  # delete and return; None if missing

.get() — the safer way to read

If you ask for a key that does not exist using bracket notation, Python throws a KeyError and your program crashes. This is the single most common dictionary bug for beginners. The fix is .get(), which returns a default value instead of crashing.

The two-argument form customer.get("phone", "N/A") is especially useful when processing records that may have optional fields — exactly the kind of thing you encounter with real-world CSV exports where some rows have data and some don't.

customer.get("email")                  # None if missing (no crash)
customer.get("email", "no email")      # "no email" if missing
customer.get("tier", "bronze")         # "bronze" if tier not set

Iterating

A plain for loop over a dictionary iterates the keys only. This surprises beginners who expect both key and value. To get both, use .items() which yields (key, value) tuples. Use .keys() to be explicit about iterating keys, and .values() to iterate only values — useful when you want to sum or aggregate the values.

The .items() pattern is the most useful and appears in virtually every real dictionary-processing script. Memorize this one.

product = {"name": "Laptop", "price": 999, "stock": 12}

# Keys only (default)
for key in product:
    print(key, product[key])

# Keys and values together
for key, val in product.items():
    print(f"{key}: {val}")

# Sum all numeric values
total = sum(product.values())  # won't work here (mixed types) — just for illustration

Lookup tables — a common business pattern

One of the most powerful and underused patterns with dictionaries is the lookup table: a dictionary that maps one value to another. Instead of a chain of if/elif statements that checks each possible tier, you define the mapping once in a dictionary and look up the result in one line. This is more readable, easier to maintain, and simpler to extend — adding a new tier is just one new line in the dictionary.

This pattern replaces what you might have done with a VLOOKUP in Excel. The dictionary is the lookup table, and .get(key, default) is the lookup operation.

tier_discounts = {
    "bronze": 0.00,
    "silver": 0.05,
    "gold":   0.10,
    "platinum": 0.15,
}

# One-line lookup — no if/elif chain needed
discount = tier_discounts.get(customer["tier"], 0)
final_price = purchase * (1 - discount)
print(f"Discount: {discount*100:.0f}%  Final: ${final_price:.2f}")

Code Walkthrough

Nested dictionary: a product catalog

Dictionaries can nest inside dictionaries to any depth. A product catalog is a natural example: the outer dictionary maps product names to records, and each record is itself a dictionary of attributes. Nested access uses multiple bracket calls in sequence.

catalog = {
    "Widget A": {"price": 9.99,  "stock": 150, "category": "supplies"},
    "Widget B": {"price": 14.99, "stock": 30,  "category": "supplies"},
    "Gadget C": {"price": 49.99, "stock": 8,   "category": "electronics"},
}

# Access nested value
print(catalog["Gadget C"]["price"])    # 49.99

# Update stock after a sale
catalog["Widget A"]["stock"] -= 5

# List all products with low stock
for name, details in catalog.items():
    if details["stock"] < 20:
        print(f"LOW STOCK: {name} — only {details['stock']} left")

List of dictionaries: processing multiple records

A list of dictionaries is the Python equivalent of a spreadsheet table — each dictionary is a row, each key is a column name. This structure is extremely common when reading CSV files (Module 12) and when working with pandas DataFrames (Unit 4). Practice it now.

customers = [
    {"name": "Alice",   "tier": "gold",     "ytd_spend": 12400},
    {"name": "Bob",     "tier": "silver",   "ytd_spend": 3200},
    {"name": "Carol",   "tier": "platinum", "ytd_spend": 28900},
    {"name": "David",   "tier": "bronze",   "ytd_spend": 450},
]

# Find the highest spender
top = max(customers, key=lambda c: c["ytd_spend"])
print(f"Top customer: {top['name']} (${top['ytd_spend']:,})")

# Total revenue from gold+ tiers
premium = [c["ytd_spend"] for c in customers if c["tier"] in ("gold", "platinum")]
print(f"Premium revenue: ${sum(premium):,}")

Common Errors

KeyError on bracket access

This is the most common dictionary error. Bracket access crashes if the key doesn't exist. Always ask: "Am I certain this key is in every record?" If there's any doubt, use .get().

# WRONG — crashes if "phone" key is missing
phone = customer["phone"]

# RIGHT — returns "N/A" instead of crashing
phone = customer.get("phone", "N/A")

Iterating keys when you want values

A plain for loop gives you keys, not key-value pairs. This surprises students who expect both. The fix is always .items().

# WRONG — prints keys, then crashes trying to print key as value
for item in product:
    print(item["price"])   # TypeError: string indices must be integers

# RIGHT
for name, details in catalog.items():
    print(name, details["price"])

Discussion Questions

  1. Your company's CRM system stores customer data in a spreadsheet with 20 columns. A junior analyst wrote a script that accesses data by column index (e.g., row[7] for the email address). Why is this fragile? How would you redesign it using dictionaries?
  2. A colleague proposes storing the product catalog as a list of tuples: [("Widget A", 9.99, 150), ("Widget B", 14.99, 30)]. What are the advantages and disadvantages of this approach compared to a list of dictionaries?
  3. When would you use a dictionary as a lookup table instead of a series of if/elif statements? Give a real business example where the dictionary approach would save significant code.
  4. Dictionaries are unordered by default (though Python 3.7+ preserves insertion order). Does order matter for a customer record? When might you want to sort the output of a dictionary, and how would you do it?

Key Takeaways
  • Dictionaries give you named access to data — record["email"] is clearer and safer than record[6].
  • Always use .get(key, default) when a key might be missing — never rely on bracket access for optional fields.
  • Use .items() when iterating if you need both key and value simultaneously.
  • A lookup-table dictionary replaces chains of if/elif and makes adding new cases trivial.
  • A list of dictionaries is the Python equivalent of a spreadsheet table — master this pattern before Module 12.
