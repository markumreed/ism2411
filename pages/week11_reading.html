Dictionaries

A customer record: {"name": "Alice", "tier": "gold", "ytd_spend": 12400}. Dictionaries are how Python represents structured real-world data.

Why this matters

A list says "here are five products in order." A dictionary says "here is one product, and here is everything I know about it." That is a fundamentally different kind of data structure — one that mirrors how businesses actually think about records. A customer is not just a position in a list; a customer has a name, an email, a loyalty tier, a year-to-date spend. A dictionary keeps all of that information together under one variable, accessible by readable labels instead of cryptic index numbers.

Consider what happens in a CRM system when you look up a customer. The system doesn't go to "row 3472, column 7" — it goes to the record for customer ID 3472 and asks for the "email" field. That is dictionary thinking. In Python, customer["email"] is far more readable and less error-prone than customer[6], which could silently break the moment anyone reorders the columns. Dictionaries give you named access, which is both clearer and safer.

Dictionaries are also the bridge to pandas DataFrames, which you'll encounter in Unit 4. A DataFrame is essentially a table where each column is a dictionary under the hood. Once you understand how to create, access, update, and iterate over a Python dictionary, you will find the pandas API intuitive because it follows the same logic. The skills you build in this module unlock everything that comes after it.

Business context

A customer record: {"name": "Alice", "tier": "gold", "ytd_spend": 12400}. Use a dictionary to store a complete customer profile. Use a lookup-table dictionary to apply tier-based discounts in a single line.

Core Concepts

Dictionary

An unordered collection of key-value pairs in curly braces. Keys are usually strings; values can be anything. customer = {"name": "Alice", "balance": 1500.00}.

Key Access

customer["name"] returns "Alice". If the key does not exist, Python raises a KeyError. Always use .get() when you're not sure a key exists.

.get() method

customer.get("email", "N/A") returns the value if the key exists, or the default "N/A" if it doesn't — no KeyError. This is the safe way to read optional fields.

Add / Update / Delete

Add: customer["email"] = "alice@co.com". Update: customer["balance"] = 2000. Delete: del customer["email"] or customer.pop("email").

Iterating

for key, value in customer.items() gives you both. for key in customer gives just keys. for value in customer.values() gives just values.

Business Scenario

You work in the loyalty department of a retail chain. The system stores each customer as a dictionary. Your job is to calculate the correct discount based on the customer's tier, apply it to their purchase, and update their year-to-date spend. Here is the full workflow in Python:

# Customer record
customer = {
    "name": "Alice",
    "email": "alice@example.com",
    "tier": "gold",
    "ytd_spend": 12400.00
}

# Discount lookup table — maps tier to discount rate
tier_discounts = {
    "bronze":   0.00,
    "silver":   0.05,
    "gold":     0.10,
    "platinum": 0.15
}

# This purchase
purchase_amount = 250.00

# Look up discount safely (default 0 if tier unknown)
discount_rate = tier_discounts.get(customer["tier"], 0)
discount = purchase_amount * discount_rate
final_price = purchase_amount - discount

# Update year-to-date spend
customer["ytd_spend"] += final_price

# Report
print(f"Customer: {customer['name']} ({customer['tier']})")
print(f"Purchase: ${purchase_amount:.2f}")
print(f"Discount ({discount_rate*100:.0f}%): -${discount:.2f}")
print(f"Final price: ${final_price:.2f}")
print(f"New YTD spend: ${customer['ytd_spend']:.2f}")

Key Vocabulary

  • Dictionary — a collection of key-value pairs enclosed in curly braces; keys must be unique and immutable
  • Key — the identifier used to look up a value in a dictionary; typically a string
  • Value — the data associated with a key; can be any Python object, including another dictionary
  • KeyError — the exception raised when you try to access a key that doesn't exist in a dictionary using bracket notation
  • .get() — a dictionary method that returns a value for a key, or a default if the key is missing, without raising an error
  • .items() — returns key-value pairs as tuples, used for iterating over both key and value simultaneously
  • Nested dictionary — a dictionary whose values are themselves dictionaries; used to represent hierarchical data like a product catalog
  • Lookup table — a dictionary used to map one value to another, replacing a chain of if/elif statements
Common Misconceptions

KeyError on access: customer["phone"] raises a KeyError if "phone" is not in the dictionary — even if the variable exists. Always use .get() for optional fields. The pattern customer.get("phone", "No phone on file") is your friend.

Dictionaries don't have order — or do they? As of Python 3.7, dictionaries maintain insertion order. But you should not rely on order for logic; dictionaries are designed for key-based lookup, not positional access. If you need a specific order, sort the keys explicitly.

Mutable default arguments: If you try to use a list or dictionary as a default parameter value in a function definition, you'll get surprising behavior — the same object is reused across all calls. Use None as the default and create the list/dict inside the function.

How do you add a key-value pair to a dictionary?
  • a) dict.add("key", val)

Which method returns all key-value pairs as tuples?
  • a) .keys()
  • b) .values()
  • c) .items()
  • d) .pairs()
Answer
c) .items()
Q7 · Code Trace
What does this code print?
catalog = {"Widget A": {"price": 9.99, "stock": 150}}
print(catalog["Widget A"]["stock"])
Answer
150 — Access the outer dict by key "Widget A", then the inner dict by key "stock"
Q8 · Bug Fix
This code crashes. Why, and how do you fix it?
customer = {"name": "Bob", "tier": "silver"}
print(customer["email"])
Answer
KeyError — "email" is not in the dictionary. Fix: use customer.get("email", "No email") instead of bracket notation.
Q9 · Multiple Choice
What is the main advantage of using a dictionary as a lookup table instead of a chain of if/elif statements?
  • a) Dictionaries are faster to type
  • b) Dictionaries are easier to extend — add a new entry without changing logic
  • c) if/elif doesn't work with strings
  • d) Dictionaries automatically sort the keys
Answer
b) Dictionaries are easier to extend — to add a new tier or code, you add one line to the dict rather than another elif branch throughout your code
