Today's walkthrough

Arithmetic operators

These work just like a calculator — with two tricky cases: // (integer division) and % (modulo / remainder). Standard division in Python 3 always produces a float: 10 / 4 gives 2.5, not 2. This is a change from Python 2 and from the behavior of division in many other languages like C or Java. If you need integer division — whole-number quotient with no decimal — use //.

Modulo is the remainder after division. 17 % 5 is 2 because 5 goes into 17 three times (15) with 2 left over. Modulo is more useful than it first appears: checking if a number is even (x % 2 == 0), finding the units left over after filling full cartons, rotating through days of the week, and many other cyclic-counting problems all use modulo. Get comfortable with it — it will show up in every domain from e-commerce to finance to scheduling.

10 + 3  # 13
10 - 3  # 7
10 * 3  # 30
10 / 3  # 3.333... (float)
10 // 3 # 3 (integer division, drops decimals)
10 % 3  # 1 (remainder)
10 ** 3 # 1000 (exponent)

Order of operations

Python follows standard math rules: parentheses first, then exponents, then * /, then + -. When in doubt, add parentheses — they make code clearer too. A common business formula mistake is writing price - cost / price when you mean (price - cost) / price. Without parentheses, the division happens first: cost/price, then price minus that. The result is completely wrong, and Python will not warn you because it is syntactically valid.

The professional habit is to always use parentheses in multi-step business formulas, even when the default precedence gives the right answer. This makes the code self-documenting — anyone reading it can see the intended calculation without having to memorize operator precedence rules. Code clarity is more important than brevity when formulas have real monetary consequences.

price = 50
discount = 0.1
tax = 0.07

# without parens — wrong order, wrong answer!
total_wrong = price - price * discount + price * tax

# with parens — clear and correct
total_right = (price * (1 - discount)) * (1 + tax)

Comparison operators

These return True or False — the booleans from Module 03. Every comparison you write this semester — "is this product's margin above the threshold?", "is this customer's balance over 90 days?", "is this order quantity above the minimum?" — is a comparison expression. The result is always a boolean, and that boolean will be used by an if statement to decide what to do next (Module 05).

The critical mistake to avoid: confusing = (assignment) with == (equality test). In Python, using = inside a condition is a SyntaxError, which is helpful — but in some other languages it is valid and causes silent bugs. Make it muscle memory: single equals assigns, double equals compares.

price == 50  # equal       (note: == NOT =)
price != 50  # not equal
price >  50  # greater than
price <= 50  # less than or equal

Logical operators

Combine comparisons with and, or, not. These let you express compound business rules: "flag a transaction if the amount is over $10,000 AND the customer account is less than 30 days old." The and operator requires both conditions to be True; or requires at least one to be True; not flips whatever follows it. These operators are evaluated after all comparisons, so x > 5 and y < 10 checks both comparisons first, then applies and.

price > 100 and quantity > 5      # both must be true
price > 100 or quantity > 5       # either is enough
not in_stock                       # flips True/False

Business formulas you will write all semester

These formulas come up in every dataset we touch. Memorize the Python equivalents. Notice that gross margin is expressed as a decimal (0.333...) unless you format it with :.1% to get the percentage string. The formula for discounted price applies discount to the original price, then multiplies: price * (1 - discount_rate). Applying tax after a discount: discounted_price * (1 + tax_rate). These patterns repeat throughout the course.

revenue      = price * quantity
gross_margin = (price - cost) / price
profit       = revenue - (cost * quantity)
discount_amt = price * 0.10
after_disc   = price * (1 - 0.10)
after_tax    = after_disc * (1 + 0.07)

print(f"Revenue:  ${revenue:,.2f}")
print(f"Margin:   {gross_margin:.1%}")
print(f"Profit:   ${profit:,.2f}")
print(f"Final:    ${after_tax:.2f}")

Detailed Code Walkthrough

A complete pricing calculator

This script asks for all inputs, computes a full set of business metrics, and prints a formatted report. Study how the formulas chain together — each output variable is the input to the next formula.

# pricing_calculator.py
product  = input("Product name: ")
price    = float(input("Retail price: $"))
cost     = float(input("Unit cost:    $"))
qty      = int(input("Units sold:   "))
disc_pct = float(input("Discount (%): ")) / 100   # convert 10 → 0.10
tax_pct  = 0.07  # fixed tax rate

revenue      = price * qty
gross_margin = (price - cost) / price
discount_amt = price * disc_pct
final_price  = (price - discount_amt) * (1 + tax_pct)
net_revenue  = final_price * qty
profit       = net_revenue - (cost * qty)

print(f"\n=== {product} Report ===")
print(f"Gross revenue:    ${revenue:>10,.2f}")
print(f"Gross margin:     {gross_margin:>10.1%}")
print(f"Discount ({disc_pct:.0%}):    ${discount_amt:>10.2f}")
print(f"Final price:      ${final_price:>10.2f}")
print(f"Net revenue:      ${net_revenue:>10,.2f}")
print(f"Net profit:       ${profit:>10,.2f}")

Using modulo for packaging logic

Floor division and modulo together solve the "how many full boxes" problem that comes up in inventory and logistics.

# packaging.py
units_ordered = int(input("Units ordered: "))
units_per_box = 12

full_boxes    = units_ordered // units_per_box   # how many complete boxes
leftover      = units_ordered % units_per_box    # how many units remain

print(f"Full boxes of {units_per_box}: {full_boxes}")
print(f"Loose units:               {leftover}")
# e.g. 100 units → 8 full boxes, 4 loose

Common Errors

Wrong order of operations in a business formula

# WRONG — division happens before subtraction due to precedence
margin_wrong = price - cost / price   # this is price - (cost/price), not (price-cost)/price

# CORRECT — parentheses enforce the intended formula
margin_right = (price - cost) / price

Type mismatch: string + number

# WRONG — user typed "5" from input(), which is a str
units = input("Units: ")     # units = "5" (str)
total = units * 49.99        # "5" * 49.99 → TypeError

# CORRECT — convert to int before multiplication
units = int(input("Units: "))
total = units * 49.99

Integer division surprise

# WRONG — expecting a percentage but getting 0 (int division behavior in Python 2 style thinking)
# In Python 3 this is NOT a problem for /, but students often expect // to round
margin = 20 // 60    # gives 0, not 0.33

# CORRECT — use / for true division
margin = 20 / 60     # gives 0.3333...

Discussion Questions

  1. A script computes margin = price - cost / price. For price=100 and cost=60, what does this formula return, and what should it return? Write the corrected version.
  2. An e-commerce company applies a 15% discount and then adds 7% tax. Write the Python expression for the final price. Does the order of operations (discount first vs. tax first) matter mathematically? Verify with numbers.
  3. A warehouse manager needs to know how many full pallets of 48 boxes can be shipped from an order of 500 boxes, and how many boxes are left over. Write the expressions using // and %.
  4. What is the difference between price > 100 and in_stock vs. price > 100 or in_stock? In what business situation would each version be appropriate?

In-class activities

  • Compute the tax-inclusive total of a $200 cart with 7% tax and a 15% discount applied first.
  • Trace this: what is the value of x % 2 when x is even? When x is odd?
  • Write a comparison expression that is True only if margin is above 30% AND price is above $50.
  • A fulfillment center packs items in boxes of 24. How many full boxes can be made from 237 items, and how many loose items remain?
  • Predict the output of print(10 / 4, 10 // 4, 10 % 4) before running it.
Key Takeaways
  • Python has seven arithmetic operators: + - * / // % ** — know them all.
  • Order of operations is PEMDAS; use parentheses to make your intent explicit in business formulas.
  • Comparison operators (== != > < >= <=) return True or False.
  • Logical operators (and, or, not) combine boolean values to express compound business rules.
  • Type mismatches ("5" + 5) cause TypeErrors — always convert input before arithmetic.
