Today's walkthrough

The five commands you will use every module

These commands move you through the file system and create new folders. They are the same on macOS, Linux, and the Windows terminal (PowerShell uses the same names too). Memorizing five commands gives you complete navigation ability across any system you will ever use — from your laptop to a cloud server running a data pipeline.

The terminal feels foreign at first because there is no visual feedback — no icons, no drag-and-drop, just text. But this is also its power: everything you can do with a GUI you can do with commands, and everything you do with commands can be automated. When you type pwd, you are running a tiny program. When you type mkdir reports, the OS creates a directory entry in its file allocation table. You are talking directly to the operating system, with no translation layer between you and the machine.

pwd          # print working directory — "where am I?"
ls           # list files in the current directory
cd folder/   # change directory into "folder"
cd ..        # go up one level
mkdir name   # make a new directory called "name"

Relative vs. absolute paths

An absolute path starts from the root of the file system ("/ " on Mac/Linux, "C:\" on Windows). A relative path starts from where you currently are. Inside a project, always use relative paths so the code works on anyone's machine. This is not just a convention — it is the difference between a script that runs correctly and one that only works on the developer's laptop.

The special notation . (single dot) means "the current directory." So ./data/sales.csv is identical to data/sales.csv — both refer to a file inside a data subdirectory of wherever you are now. The notation .. (two dots) means "the parent directory." So ../reports/ means "go up one level, then into reports." You can chain these: ../../config/settings.json goes up two levels then into config.

# absolute (machine-specific):
/Users/alice/projects/sales/data.csv

# relative (portable):
data/sales.csv

# relative with parent traversal:
../data/sales.csv

Running a Python script

Once Python and VS Code are installed, you write a .py file, save it, then run it from the terminal in the same folder. The key habit to build here is always running scripts from the directory that contains them — this ensures relative paths in your code resolve correctly. If your script opens data/sales.csv, that path is interpreted relative to the directory you ran the script from, not the directory the script is stored in.

You can confirm Python is installed with python3 --version. If the command is not found, follow the setup guide in the course resources. You should see something like Python 3.11.4. The 3 matters — Python 2 is no longer supported and behaves differently in important ways.

$ cd ~/ism2411/module02
$ ls
hello.py
$ python3 hello.py
Hello, ISM2411!

What .py files are

A .py file is just a text file with Python code in it. The .py extension tells your editor to highlight Python syntax and tells the interpreter what to do with it. The file itself is plain text — you could open it in Notepad. This matters because it means Python scripts are easy to share (just send the file), easy to version control (Git tracks text changes line by line), and easy to read on any machine without special software.

The Python interpreter reads your .py file from top to bottom, executing each statement in order. If it encounters a syntax error — a typo, a missing colon, an unclosed parenthesis — it stops and prints an error message with the line number. Reading error messages is a skill. The last line of a Python traceback tells you what went wrong; the lines above show you where in the call stack the error occurred.

Detailed Code Walkthrough

Creating and running your first script

Open VS Code, create a new file, save it as hello.py. Type the following. Then open a terminal, navigate to the folder, and run it.

# hello.py — your first Python script
# The '#' character starts a comment. Python ignores everything after it.

print("Hello, ISM2411!")     # sends text to the terminal
print("This is Module 02")   # you can call print() as many times as you like

A script that builds a folder structure

This is closer to real automation. The os module gives you access to file system operations from inside Python — the same operations you were doing manually with mkdir and cd.

# setup_project.py
# Creates a standard project folder structure
import os

folders = ["data", "reports", "scripts", "notes"]
base = "ism2411_project"

os.makedirs(base, exist_ok=True)   # create the base folder
for folder in folders:
    path = os.path.join(base, folder)
    os.makedirs(path, exist_ok=True)
    print(f"Created: {path}")

print("Project structure ready.")

Common Errors

Error: "No such file or directory"

This almost always means you are in the wrong directory when you run your script, or the relative path in your script is wrong. Check with pwd and ls before running.

# WRONG — running from the wrong directory
$ pwd
/home/alice
$ python3 module02/hello.py
# If hello.py tries to open "data/sales.csv", Python looks for
# /home/alice/data/sales.csv — not /home/alice/module02/data/sales.csv

# CORRECT — navigate to the script's directory first
$ cd module02
$ python3 hello.py

Error: "command not found: python3"

Python is not installed, or it is installed but not on your system PATH. Try python --version (no 3). If that works, use python instead of python3. If neither works, install Python from python.org and restart your terminal.

# Check which python commands are available:
$ python3 --version
$ python --version
$ which python3    # shows the install path on Mac/Linux

Key Takeaways
  • pwd, ls, cd, mkdir — five commands that give you complete file system navigation.
  • Absolute paths start from root and work from anywhere; relative paths start from your current location and are more portable.
  • Always cd into your project folder before running a script to ensure relative paths resolve correctly.
  • A .py file is plain text — readable and shareable without special software.
  • The terminal is the universal interface for all programming environments — it looks the same on every machine and never changes.
